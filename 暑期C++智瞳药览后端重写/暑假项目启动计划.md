# 智瞳药览 - C++后端重写项目启动计划

---

## 🎯 项目目标

用C++重写智瞳药览AI后端，支持小程序前端调用，实现药盒图片识别和AI总结功能。

---

## 📊 现有代码分析

### 当前架构（Python/Flask）
| 模块 | 文件 | 核心功能 |
|------|------|----------|
| 入口 | `app.py` | Flask应用初始化，CORS配置 |
| 路由 | `routes/analyze.py` | 接收图片上传，调用AI分析 |
| AI模块 | `ai/analyzer.py` | DeepSeek API调用，JSON解析 |
| OCR模块 | `ocr/scanner.py` | PaddleOCR本地文字识别 |

### 关键发现
1. **图片处理**：当前是**文件上传**，不是URL方式
2. **OCR方案**：使用本地PaddleOCR（Python）
3. **LLM方案**：调用DeepSeek API
4. **数据库**：当前无数据库，无历史记录功能

## 🛠️ 技术栈确认

| 模块 | 技术选择 | 版本 | 选型理由 |
|------|----------|------|----------|
| HTTP服务 | cpp-httplib | v0.12.0+ | header-only、零依赖、支持文件上传 |
| JSON处理 | nlohmann/json | v3.11.2+ | 语法接近Python、学习成本低 |
| 数据库 | SQLiteCpp | v3.3.1+ | RAII封装、异常安全、轻量嵌入式 |
| 网络请求 | cpp-httplib | - | 内置HTTP客户端，调用云端API |

## 📋 核心功能列表（MVP）

### 基础功能
1. **接收药盒图片上传** - 接收小程序上传的药盒图片文件
2. **调用云端OCR API** - 识别药盒文字（替代本地PaddleOCR）
3. **AI总结生成** - 调用DeepSeek进行药品信息总结
4. **JSON响应** - 返回结构化的识别结果给小程序
5. **历史记录存储** - 将查询记录持久化到SQLite数据库

### API接口设计（兼容前端）
| 接口路径 | HTTP方法 | 功能描述 | 与原接口对比 |
|----------|----------|----------|--------------|
| `/analyze` | POST | 上传图片，返回分析结果 | **兼容原接口** |
| `/history/list` | GET | 查询用户历史记录列表 | **新增** |
| `/history/detail` | GET | 查询单条历史记录详情 | **新增** |
| `/health` | GET | 健康检查接口 | **新增** |

## 🚨 风险预案

### 优先级排序（从高到低）
1. **核心功能**：图片上传 + OCR识别 + AI总结（必须保留）
2. **基础功能**：HTTP服务、JSON响应（必须保留）
3. **增强功能**：历史记录、用户管理（可降级）
4. **优化功能**：并发处理、日志系统（可延后）

### 可降级方案
| 功能 | 降级方案 | 影响评估 |
|------|----------|----------|
| SQLite数据库 | 改用JSON文件存储 | 失去并发访问和结构化查询能力 |
| 用户管理 | 简化为单用户模式 | 不支持多用户隔离 |
| 历史记录列表 | 只保留最近N条记录 | 查询范围受限 |
| 健康检查 | 移除或简化 | 运维监控能力下降 |

### 最坏情况最小可行版本
- 仅保留 `/analyze` 核心接口
- 使用文件存储替代数据库（每次写入追加到JSON文件）
- 不做用户认证和权限控制
