# 暑假项目技术选型

---

## 📋 初步技术栈决策

**cpp-httplib + nlohmann/json + SQLiteCpp**

---

## 🛠️ 核心库选型

### 1. cpp-httplib
**一句话优势**：零依赖、零编译配置，只需 `#include "httplib.h"` 即可使用的header-only HTTP库。

**详细说明**：
- 不需要单独编译库文件，不需要配置CMake链接
- 通用性极强，所有C++项目均使用同一头文件
- 对Python开发者友好，接口风格与Python HTTP库类似

---

### 2. nlohmann/json
**一句话优势**：让C++的JSON操作像Python一样自然的header-only库。

**详细说明**：
- 设计哲学是"让C++的JSON操作像Python一样自然"
- 写过 `json.loads()` 的人看到 `nlohmann::json::parse()` 会觉得很熟悉
- 简洁高效，最少只需要三行即可组成完整的JSON字段

---

### 3. SQLiteCpp
**一句话优势**：将C风格SQLite API封装成RAII对象的现代C++库。

**详细说明**：
- 把C风格的手动 `sqlite3_open`/`sqlite3_exec`/`sqlite3_close` 封装成RAII对象
- 构造时自动打开，析构时自动关闭
- 异常用 `try/catch` 处理，无需手动检查返回码

---

## 🎯 选型理由

| 库 | 选型理由 |
|----|----------|
| **cpp-httplib** | 零配置、跨平台、易于集成，适合快速搭建HTTP服务 |
| **nlohmann/json** | 语法自然、学习成本低、性能优秀，适合处理JSON数据 |
| **SQLiteCpp** | RAII设计、异常安全、接口清晰，适合轻量级数据库操作 |

---

**核心价值**：这套技术栈兼顾了开发效率和运行性能，非常适合暑假项目的快速开发和部署。

## 📊 详细技术对比

### 1. HTTP库对比：cpp-httplib vs Boost.Beast vs Crow

**三个库定位不同**：
- **cpp-httplib**：轻量 HTTP 客户端/服务端库
- **Boost.Beast**：基于 Boost.Asio 的低级 HTTP/WebSocket 库
- **Crow**：类似 Flask 的 Web 微框架

**选择 cpp-httplib 的核心理由**：

- **header-only**：单头文件即可使用，零编译依赖，直接 `#include` 就行，极大简化项目构建
- **零外部依赖**：不强制依赖 OpenSSL、Boost 或 zlib，可选性很高
- **API 简单直观**：发起 GET/POST 请求仅需几行代码，支持同步与异步，对调用云端 OCR 和 AI 问答 API 绰绰有余
- **专为小型项目设计**：没有 Beast 那种模板元编程的复杂度，也没有 Crow 引入的完整路由、中间件系统

**对比结论**：
- Boost.Beast 需要整个 Boost 库，编译慢、体量大
- Crow 是服务器框架，用在客户端属于杀鸡用牛刀
- cpp-httplib 是配合小型客户端工具的最佳匹配

---

### 2. JSON库对比：nlohmann/json vs RapidJSON

**为什么选 nlohmann/json？**

| 维度 | nlohmann/json | RapidJSON |
|------|--------------|-----------|
| **语法友好度** | ⭐⭐⭐⭐⭐ 极度友好，像 Python json 库 | ⭐⭐⭐ 繁琐，SAX/DOM 接口复杂 |
| **学习成本** | 几乎为零 | 需要学习显式内存分配器和 Value 类型操作 |
| **开发效率** | 高 | 中 |
| **性能** | 对几 KB 数据足够（微秒级） | 更高 |
| **适用场景** | 数据量小、追求开发效率 | 超大数据量、性能敏感 |

**结论**：当性能差异不影响用户体验时，永远选择表达能力更强的库。项目数据量不大，选 nlohmann/json。

---

### 3. 数据库方案：SQLite vs 文本文件

**为什么数据库比文件好？**

**① 并发访问**
- SQLite 提供数据库级别的并发控制（WAL 模式允许一个写者和多个读者），保证数据不会因竞争而损坏
- 原始文件需要自己实现文件锁，很容易出现写入覆盖或数据截断

**② 结构化查询**
- 想按日期查看记录？想搜索包含某个关键词的识别结果？
- SQL 一句搞定：`SELECT * FROM records WHERE content LIKE '%关键词%' ORDER BY date DESC`
- 文件存储需要自己设计行格式、解析、遍历、排序，代码量暴增且容易出错

**附加优势**：
- SQLite 是嵌入式的，零部署
- 通过 SQLiteCpp 封装，写出现代 C++ 风格的代码（RAII、异常安全）
- 资源占用极小

---

### 4. 初步决策

**我选择 cpp-httplib + nlohmann/json + SQLiteCpp 作为暑假项目的技术栈，理由是**：三者均为 header-only 或极轻量库，零额外构建负担；HTTP 客户端调用 API、JSON 解析返回数据、SQLite 持久化记录，各司其职且接口都简洁直观，能以最小的心智代价快速完成功能闭环，让你专注于业务逻辑和 AI 交互，非常适合暑假小型项目的敏捷开发。
